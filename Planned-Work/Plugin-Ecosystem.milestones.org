* Introduction
  :PROPERTIES:
  :initiative_goals: Provide a pluggable contract so consumers can swap among open-source and commercial talking-head/lip-sync providers without changing GuiAssert itself, validated by strong integration tests per plugin (real subprocess invocations or HTTP mock-server tests with exact request-body assertions; live tests gated for credentialed runs).
  :related_specs: [[file:../README.md][GuiAssert README]], [[file:../src/gui_assert/talking_head/core.nim][TalkingHeadProvider contract]]
  :END:

  GuiAssert defines the `TalkingHeadProvider` contract and ships only
  lightweight built-ins (`stock_avatar`). Heavy providers — local ML
  models or commercial APIs — live in sibling repositories named
  `GuiAssert-<Provider>` and are imported by consumers behind a
  `-d:with<Provider>` compile gate.

** Plugin contract
   The contract is `GuiAssert/src/gui_assert/talking_head/core.nim`:

   - `TalkingHeadProvider` value-type with `name`, `isAvailable`, `generate`.
   - `TalkingHeadRegistry` for runtime dispatch by string name.
   - `cacheKeyFor(avatar, narration, providerName, device)` — stable hash.
   - `applyCache(cacheDir, key, outputMp4, generator)` — generic memoisation.
   - `optsFromMetadata(meta, scriptDir)` — translates YAML
     `metadata.talking_head:` blocks into `TalkingHeadOpts`.

   Each plugin exports a `register<Plugin>(reg: TalkingHeadRegistry)` proc
   plus a constructor `<plugin>Provider(): TalkingHeadProvider`.

    - Apple Silicon required compiling full =mmcv==2.1.0= from source:
      there are no pre-built arm64 wheels for the =mmcv._ext=
      extension, and =mmcv-lite= is not sufficient because mmpose's
      =EDPoseHead= imports =MultiScaleDeformableAttention= from
      =mmcv.ops= at module-import time. The install script builds
      mmcv with =MMCV_WITH_OPS=1 FORCE_CUDA=0 MAX_JOBS=2= against the
      older =MacOSX15.sdk= libc++ (the macOS 26 SDK's
      =_LIBCPP_NO_SPECIALIZATIONS= annotations break PyTorch 2.2's
      =c10::util::strong_type= specialisations). Build time ~3-6 min.
    - Total install time on a clean machine is ~30-40 minutes,
      dominated by 4.3 GB of model-weight downloads from Hugging Face
      (musetalkV15 UNet, musetalk v1.0 UNet, sd-vae, whisper-tiny,
      DWPose, face-parse-bisent). The install script idempotently
      skips already-downloaded files on re-run.
    - Live render time on an M-series Mac with MPS: ~19 s wall-clock
      for a 3.5 s narration (87 frames at 25 FPS, H.264 + AAC,
      400x400, ~87 KB output). The =PYTORCH_ENABLE_MPS_FALLBACK=1=
      env var (set by the wrapper) covers MPS-unsupported aten ops in
      the diffusers VAE + mmpose RTMPose backbones.
    - The install script's =apply_patch= helper uses a
      whitespace-tolerant regex match (trailing spaces on any needle
      line are ignored). The upstream
      =musetalk/utils/face_parsing/__init__.py= ships with a stray
      trailing space after =net.load_state_dict(torch.load(model_pth))=
      — an earlier strict =grep -qF= match silently no-oped on that
      patch and the unpatched branch later =FileNotFoundError='d on
      =./models/face-parse-bisent/79999_iter.pth=.
    - The face-parse-bisent weight (=79999_iter.pth=) is mirrored from
      the =camenduru/MuseTalk= Hugging Face repo (auth-free). Upstream's
      original Google Drive ID + =gdown= path is kept as a last-resort
      fallback; the =ByteDance/LatentSync= auxiliary tree used by
      earlier revisions has been deleted upstream and now 404s.

*** Implementation Notes
    - The =std/httpclient= keep-alive behaviour clashes with
      =std/asynchttpserver=: after the server closes the response
      socket, httpclient's cached connection is half-dead and the
      next request times out or fails =ProtocolError=. The plugin
      forces =Connection: close= on every request and re-builds a
      fresh =HttpClient= per API call (via a =withFreshClient=
      template). This is also a small win against real-world
      upstream proxies that drop idle sockets during the polling
      sleep gap.
    - The mock-server test drives =asyncdispatch.poll()= on a
      dedicated thread (=createThread=), so the main thread can
      block on synchronous httpclient calls. =--threads:on= is
      therefore required for the test build.
    - =/images= + =/audios= are uploaded via =MultipartData=. The
      mock's request-trace assertion verifies both the
      =multipart/form-data= Content-Type and the verbatim portrait /
      narration bytes appear in the request body.
    - The bundled =tests/fixtures/portrait.png= is the same
      PD-US-expired Einstein crop used by sibling Wav2Lip / MuseTalk
      fixtures, for consistency.
    - The narration WAV fixture (=tests/fixtures/narration.wav=,
      ~113 KB, 3.55 s, 16 kHz mono) is generated once via =say= +
      =ffmpeg= and committed so the tests are self-contained.

*** Implementation Notes
    - The =TalkingHeadProvider.generate(narrationWav, outputMp4, opts)=
      contract expects audio to arrive as a pre-rendered WAV. HeyGen
      v2's =video/generate= endpoint does NOT accept uploaded audio:
      it takes an =input_text= string plus a =voice_id= and synthesises
      the voiceover server-side. The plugin therefore ignores
      =narrationWav= for the API call and requires
      =opts.providerSettings["input_text"]= to be set — raising
      =TalkingHeadError= when missing. The WAV is still hashed into
      the cache key so per-session uniqueness in the WAV is preserved.
      This deviation is documented prominently in
      =GuiAssert-HeyGen/README.md=. A follow-up could speak HeyGen's
      =audio_url= mode (upload via =/v1/upload.url= first, then
      reference) to honour the WAV verbatim.
    - =cacheKeyFor='s signature (avatar + narration + provider +
      device) is fixed by the GuiAssert contract. For HeyGen the
      "avatar" is purely server-side (an =avatar_id= string) and the
      narration text is also server-side (an =input_text= string),
      so the plugin folds the HeyGen-specific knobs (=input_text=,
      =avatar_id=, =voice_id=, =width=, =height=) into the device
      slot via =heygenCacheSalt= (a SHA-1 prefix of the combined
      string). Two different =input_text= values for the same WAV
      therefore produce different cache entries — exercised by the
      mock-server test.
    - HeyGen wraps every JSON response in =({"data": ..., "code":
      100, "message": "Success"})=. The plugin's =unwrapData= helper
      strips the envelope and surfaces the inner =data= object to
      callers, so the rest of the code reads =video_id= / =status= /
      =video_url= directly without re-implementing the unwrap at each
      call site.
    - The =/result.mp4= download URL is served from HeyGen's CDN and
      does not require authentication. The mock-server test pins this
      behaviour: =X-Api-Key= must be present on every =/v2/= and
      =/v1/= request but ABSENT on the CDN download. This guards
      against accidental credential leakage to the CDN if a future
      refactor pulls the download into the main =HttpClient= path.
    - Same =std/asynchttpserver= keep-alive workaround documented in
      M3 applies: a dedicated polling thread drives
      =asyncdispatch.poll()= while the main thread blocks in
      synchronous =httpclient= calls; =--threads:on= is therefore
      required for the test build, and every API call opens a fresh
      =HttpClient= via the =withFreshClient= template.

      =GuiAssert-Synthesia/README.md=. Same shape as the M4 HeyGen
      deviation.
    - Synthesia's =Authorization= header is quirky: it carries the
      RAW API key with no =Bearer= or =Basic= prefix. This is unusual
      (HeyGen uses =X-Api-Key=; D-ID uses HTTP Basic; most REST APIs
      follow RFC 6750 =Bearer <token>=). The pure tests include
      explicit regression guards against accidental "fixes" that would
      wrap the key in a prefix.
    - Synthesia's JSON request shape uses camelCase =scriptText=
      inside the =input[]= array — NOT snake_case like the rest of
      the ecosystem (and the rest of Synthesia's own request body,
      which uses lowercase singletons =avatar= / =background= /
      =title= / =test=). The pure tests pin this with an explicit
      assertion that =script_text= is absent from the emitted JSON.
    - Synthesia's JSON response is a FLAT object — no =({"data": ...,
      "code": 100})= envelope like HeyGen. The plugin reads =id= /
      =status= / =download= directly off the top-level object.
    - =cacheKeyFor='s signature (avatar + narration + provider +
      device) is fixed by the GuiAssert contract. For Synthesia the
      "avatar" is purely server-side (an avatar string) and the
      narration text is also server-side (a =script_text= string), so
      the plugin folds the Synthesia-specific knobs (=script_text=,
      =avatar=, =background=, =title=, =test=) into the device slot
      via =synthesiaCacheSalt= (a SHA-1 prefix of the combined string).
      Two different =script_text= values for the same WAV therefore
      produce different cache entries — exercised by the mock-server
      test.
    - The =/result.mp4= download URL is served from Synthesia's CDN
      and does not require authentication. The mock-server test pins
      this behaviour: =Authorization= must be present on every =/v2/=
      request but ABSENT on the CDN download. This guards against
      accidental credential leakage to the CDN if a future refactor
      pulls the download into the main =HttpClient= path.
    - Polling cadence defaults to 10 s with a 15-minute timeout —
      more generous than the M3 D-ID (3 s / 5 min) and M4 HeyGen
      (5 s / 10 min) defaults because Synthesia renders are reliably
      the slowest of the commercial pack.
    - The =test: true= flag (the default) opts into Synthesia's
      watermarked sandbox render, which does NOT count against the
      account's monthly minute quota. The mock-server test exercises
      this path; production callers can flip =test= to false to spend
      real quota.
    - Same =std/asynchttpserver= keep-alive workaround documented in
      M3 / M4 applies: a dedicated polling thread drives
      =asyncdispatch.poll()= while the main thread blocks in
      synchronous =httpclient= calls; =--threads:on= is therefore
      required for the test build, and every API call opens a fresh
      =HttpClient= via the =withFreshClient= template.

      "code": 100})= envelope like HeyGen. The plugin reads
      =video_id= / =status= / =download_url= directly off the
      top-level object.
    - Tavus's terminal statuses are =ready= (success), =error=
      (failure), and =deleted= (treated as failure). Intermediate
